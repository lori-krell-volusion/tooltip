StyleGuide

Voice

Use active voice unless:
• passive voice is the only or best way to avoid an awkward or wordy construction
• the subject is unknown
• the emphasis should be on the receiver of the action rather than the doer
• specifying the user as the doer might seem condescending or blaming (especially in error messages)

Avoid metaphor and anthropomorphism whenever possible.

Do not make assumptions about how the user is feeling. If you think the user may have a particular concern at a specific moment, address it with an "if" statement.

Do not try to influence the reader's feelings about the product by describing actions or procedures as simple, easy, hard, difficult, or complex, or say "you only have to…" or "you simply have to…" List the required steps and let the user decide how to feel about them.

Do not use words and phrases that apologize for aspects of the software, such as unfortunately.

Do not volunteer information about what the software cannot do; simply relate how to do what the software can do.

When writing numbers, use words for 0-9 and numerals for numbers with two or more digits. Try not to begin sentences with numerals.

Internationalization

Use short, simple sentences whenever possible.

Avoid constructions with multiple and, or, or but clauses whenever possible.

When referring to time, use a 24-hour clock (do not use PM).

Use international date format (for example: December 13, 2014).

Use given name instead of first name and surname instead of last name.

Do not use as or since when you mean because.

Do not use e.g. when you mean for example or i.e. when you mean that is.

Navigation

Use click for pushing buttons, opening menus, executing commands in menus, selecting tabs, and choosing options in tabs.

Use right-click for actions that require the secondary mouse button.

Use select for checking check boxes and clear for un-checking them.

Use point to rather than hover over.

Refer to the mouse icon as the pointer.

When referring to the location of a tab option, use "on the [Name] tab."

When it will not introduce reader confusion, you can simplify the description of a navigational path or series of clicks by using the > character. For example, "Go to Settings > General Settings > Notifications and locate the Reply-to e-mail field."

Terminology

Call a menu that drops down a drop-down menu.

Do not use other adjectives to describe menus, such as pop-out or cascading, unless the context of the instruction requires a description of physical appearance.

Call a window that pops up a dialog box.

Call the private side of an account the admin area and the public side the website.

Call commands that the user cannot click unavailable rather than grayed out or dimmed.

Call text that contains a warning a message, not a warning or a warning message.

Call the primary navigation element of Mozu the Mega Menu.

Call the button you click to expose the Mega Menu the Mega Menu icon.

Use turn off and turn on for features, not enable and disable.

Instruct a user to type text in a text field.

Use create when instructing a user to generate a new entity, and add when instructing a user to apply an existing entity to another (for example, "add a product type to your product." Do not use new when instructing a user to create an entity.

Tell users to type in text boxes, and enter an element such as a file name that they can either type or select from a list, such as when selecting a file for upload.

When referring to a drop-down menu that exposes options when you click it, call it "the [Name] list."

Use drag rather than click and drag or drag and drop. You can use the adjective drag-and-drop.

Use under when referring to the position of an element below another rather than below or beneath.

Use on when referring to an element contained within a tab or menu.

Use in when referring to an element that is part of a group (for example, "in the [Name] section."

A menu button is a button with an arrow that exposes a menu whether you click the button or the arrow.

A split button is a button with an arrow that executes an action if you click the button, but exposes a menu if you click the arrow.

Style

Use a the exact text of a message when referring to it. For example: "If you receive the This file has been modified outside of the source editor. Do you want to reload it? message, click Yes."

If a message is excessively long, you can truncate or summarize it.

Use bold formatting for button names, menu names, commands or selections in menus, section headings, labels, tab names, options in tabs, and field names.

Follow the capitalization of the UI exactly, unless it is inconsistent, or uses all capitals, or uses all lower-case. In these situations, use title case for menu names, command names, command button names, dialog box titles, and tab names; and use sentence case for dialog box and page elements.

Do not capitalize toolbar, button, menu, scroll bar, and icon when naming UI elements, unless that word is part of the name.

Capitalize names of features and pages, but do not use bold styling unless you are referring to a label click. Do not capitalize the word feature or page in these cases.

Always use "the [Name] check box" when referring to check boxes.

Write text that the user should enter in text fields in bold formatting, unless you are using placeholder text such as "password." In this case, use italic formatting.

Use a comma after the penultimate item in a series.

Do not use contractions.

Do not use possessive case when referring to UI elements, trade names, or brand elements ("Mozu's").

Write the steps of a procedure in the order the user encounters them. Avoid constructions that name a UI element (such as a tab or menu) after naming an item the element contains (such as a command or option).

Use a different font or style for code excerpts and URLs. If the style sheet for the resource does not offer a preset styling option, use Courier New and precede and follow the text with a line break.

In procedures, do not include supplementary information, such as special cases or non-essential behaviors. If supplementary information is necessary, include it in a single paragraph after the procedure. If you must include it in the procedure, use expanders/togglers when available. When unavailable, include the information after the step it explains and indent it to align with the procedure text. If it includes steps, make it a separate procedure and provide a cross-reference.

Avoid explicit descriptions of system responses. If necessary to orient the user, describe the system response in the step before the system response happens or in the step immediately following the system response.

Avoid describing system responses in procedures. If necessary to orient the user, describe the system response in the step before the system response happens or in the step immediately following the system response.

If a command or dialog box option includes a colon or ellipses at the end, do not include the punctuation.

Do not use the words the and button when referring to buttons; simply use the button text in bold.

When instructing a user to expose the menu below a menu button or a split button, say “click the [Button Label] arrow.”
